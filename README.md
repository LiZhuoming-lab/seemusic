# 音乐分析平台（音频 + 乐谱）

一个面向声景音乐、频谱音乐、传统作品与当代作品分析的本地研究工具。它的目标不是替代音乐学判断，而是先帮研究者把“哪里值得重点听、重点看、重点写”自动筛出来，并逐步把音频分析与乐谱分析纳入同一个工作台。

## 项目定位

这个项目是一个面向研究与论文写作场景的 `MVP`。它把音频分析、事件检测、局部试听、交互式频谱查看和人工标注整理到同一个工作流里，适合用来辅助以下工作：

- 声景音乐结构分析
- 频谱音乐与当代音乐作品研究
- 混合媒介作品中的材料进入、音色突变与结构边界初筛
- 基于时间轴、频谱与听觉证据的论文写作整理

## 核心能力

- 支持双工作空间：
  - `音频 / 频谱分析`
  - `乐谱 / 符号分析`
- 支持 `WAV`、`FLAC`、`AIFF`、`MP3`、`M4A`
- 支持 `MusicXML`、`MXL`、`MIDI`
- 支持 `Kern / Humdrum (.krn / .kern)`
- 自动提取 `RMS`、`spectral centroid`、`rolloff`、`flatness`、`spectral flux`、`onset strength`、`novelty`
- 自动检测疑似“新材料进入 / 音色突变 / 结构变化”事件点
- 支持“传统结构模式 / 音色・声景模式”等预设参数切换
- 提供交互式总览、局部试听、局部波形与局部频谱查看
- 支持人工标签修订、备注补充、事件筛选与结构化导出
- 支持音高统计、音程序类统计、和声切片、Roman numeral 候选与主题再现候选
- 支持在右侧工作台中直接调用 `When-in-Rome` GitHub 语料库中的现成乐谱文件
- 支持在右侧工作台中直接调用 `Beethoven Piano Sonatas` GitHub 语料库中的 `kern` 乐谱
- 支持 `完满终止候选 / 半终止候选` 的第一阶段自动检测
- 同时提供图形界面与命令行入口

## 最新更新

本次更新把项目从单一的声音频谱事件分析工具，进一步扩展为一个并列的音乐分析平台。当前前端已经形成两条并列工作流：

- 左侧为原有的**声音频谱事件分析工作台**
- 右侧为新增的**music21 符号分析工作台**

本次版本的主要更新包括：

- 新增 `music21` 并列工作台  
  在保留原有音频 / 频谱分析功能的基础上，加入基于 `music21` 的符号分析入口，用于处理 `MusicXML`、`MXL`、`MIDI` 等乐谱文件，并支持音高、音级、音程、和声切片与主题 / 动机再现的初步分析。

- 接入 GitHub 开源项目 `music21`  
  右侧符号分析工作台基于 [music21](https://github.com/cuthbertLab/music21) 的能力进行封装，但不是直接展示库函数，而是将其组织为更适合音乐分析场景的工作流界面。

- 接入 GitHub 开源语料库 `When-in-Rome`  
  右侧工作台新增 [When-in-Rome](https://github.com/MarkGotham/When-in-Rome) 语料库入口，用户可以直接调用仓库中的现成乐谱文件，而不必先手动下载到本地。当前已支持按类别、作曲家和关键词筛选语料，并直接进入符号分析流程。

- 接入 GitHub 开源语料库 `Beethoven Piano Sonatas`  
  右侧工作台新增 [Beethoven Piano Sonatas](https://github.com/craigsapp/beethoven-piano-sonatas) 语料库入口，用户可以直接调用贝多芬钢琴奏鸣曲 `kern / .krn` 文件，进入当前的 `music21` 符号分析流程。

- 新增 `PAC / 半终止` 终止候选检测  
  右侧工作台现已支持第一阶段的终止分析，当前可输出：
  - `完满终止候选`
  - `半终止候选`

  这一部分采用了更贴近传统作品分析的规则：优先参考主旋律骨架、局部和声到达、前后小节低音支撑与终止窗口收束，而不是只按单一时间切片作硬判断。

- 修复按小节音高事件统计异常  
  针对部分 `kern` 语料中由解析导致的小节号错位问题，现已改为基于时间线重新映射音高事件所属小节，从而修复了“单一小节音符数异常飙高”的统计错误。

- 新增乐谱文件导出  
  在右侧工作台中，除了分析结果导出外，还支持直接导出原始 `MXL / XML` 文件，以及统一导出的 `MusicXML` 文件，方便继续在其他乐谱软件或音频软件中查看。

- 修复空和弦导致的分析报错  
  针对部分语料中可能出现的空和弦或无有效音高对象，已增加安全跳过逻辑，避免符号分析流程因单个异常对象中断。

- 优化导出区结构  
  右侧工作台的导出栏现已区分为：
  - `分析结果导出`
  - `乐谱文件导出`

  这样更方便区分“分析数据”和“原始 / 可继续使用的乐谱文件”。

## 快速开始

### 1. 安装依赖

```bash
python3 -m pip install -r requirements.txt
```

### 2. 启动图形界面

```bash
streamlit run app.py
```

启动后，在浏览器里打开终端显示的本地地址，上传音频即可开始分析。
现在在同一个界面里也可以切换到“乐谱 / 符号分析”工作空间，上传 `MusicXML / MXL / MIDI / KRN` 开始做第一阶段的和声、音高、终止与主题分析。

如果你希望直接调用公开语料，也可以在右侧工作台中切换到：

- `When-in-Rome 语料库`
- `Beethoven Piano Sonatas`

不必先手动下载乐谱。

### 2.1 本地访问（localhost）

默认情况下，Streamlit 会在本机启动一个本地网页服务，通常可以直接打开：

- `http://localhost:8501`

如果浏览器没有自动弹出，可以手动把这个地址粘贴到浏览器中打开。

如果想让同一局域网中的其他设备访问，可以使用终端中显示的 `Network URL`，通常形如：

- `http://你的局域网IP:8501`

### 3. 命令行使用

```bash
python3 -m spectral_tool.cli your_audio.wav
```

也可以带参数运行：

```bash
python3 -m spectral_tool.cli your_audio.wav \
  --channel mix \
  --target-sr 22050 \
  --n-fft 4096 \
  --hop-length 1024 \
  --min-event-distance 5 \
  --threshold-sigma 1.0
```

## 输出内容

运行后可得到：

- 事件表 `events.csv`
- 段落建议 `sections.csv`
- 逐帧特征表 `feature_table.csv`
- 结构化结果 `analysis.json`
- 文字摘要 `summary.txt`
- 波形图、频谱图、新颖度曲线、事件密度图

## 分析工作流

当前平台包含两条互补路径：

1. 音频路径：自动发现谱形变化、音色变化与结构边界
2. 乐谱路径：提取音高、和声、音程与主题再现候选

它仍然更接近“为音乐学判断提供结构化证据”，而不是“自动替代音乐分析本身”。

典型工作流如下：

1. 将音频转换为短时频谱表示
2. 提取能量、频谱重心、滚降、平坦度、谱流量、起音强度等特征
3. 组合为新颖度曲线
4. 在新颖度曲线上做峰值检测，生成候选事件点
5. 比较事件前后窗口的状态差异
6. 输出候选标签，并允许研究者继续人工修订

因此，它更适合帮助回答这些问题：

- 哪里发生了变化
- 哪里值得重点听和重点看
- 哪些时间点可能对应新材料、音色转换或结构边界

## 推荐使用方式

如果你在分析类似《六季》第一乐章《新冰》这样的作品，可以参考下面的节奏：

1. 先做一次全曲粗筛，获取事件时间轴
2. 再回到关键事件点做局部试听与局部频谱核对
3. 将自动事件与人工听觉记录对照
4. 最后整理为论文中的“时间 - 材料 - 结构功能”表

## 当前边界

- 机器能够发现声学变化，但不能自动替代音乐学解释
- 自动标签属于辅助描述，不等于最终论文结论
- 对于长时段、极弱动态或高度细腻的纹理，仍然需要人工复核和参数调整
